تایم‌اوت و Deadline در gRPC و پایتون

July 8, 2026

مهدی:

سلام. من یک مهندس نرم افزار جونیور هستم. من نمی‌دانم تایم اوت چطوری هندل میشه. من روی پایتون کد می‌زنم و شرکتمون هم grpc هست. لطفا پرکتیس‌ها و پترن‌های استفاده از تایم اوت رو بهم یاد بده.

جمینای:

سلام! به دنیای جذاب (و گاهی چالش‌برانگیز) میکروسرویس‌ها و gRPC خوش آمدی. پرسیدن این سوال نشانه بسیار خوبی است، چون خیلی از برنامه‌نویس‌ها تازه زمانی به فکر Timeout می‌افتند که کل سیستمشان در پروداکشن دچار اختلال (Cascading Failure) شده است!

در gRPC، مفهومی که با آن سر و کار داریم بیشتر به نام Deadline (مهلت زمانی) شناخته می‌شود. Timeout می‌گوید “۵ ثانیه صبر کن”، اما Deadline می‌گوید “تا ساعت ۱۰:۰۵:۰۰ کار را تمام کن”. gRPC به صورت خودکار Timeout ها را به Deadline تبدیل می‌کند.

در ادامه، نحوه پیاده‌سازی و سپس بهترین الگوها (Patterns) را با زبان پایتون برایت توضیح می‌دهم.

۱. نحوه پیاده‌سازی پایه در پایتون (gRPC)#

سمت کلاینت (Client-Side)#

وقتی درخواستی می‌فرستی، باید تعیین کنی چقدر حاضری منتظر بمانی. اگر زمان تمام شود، gRPC یک خطای DEADLINE_EXCEEDED برمی‌گرداند.

import grpc

def call_server(stub):
    try:
        # تنظیم تایم‌اوت روی ۲.۵ ثانیه
        response = stub.MyMethod(request, timeout=2.5)
        print("Response received:", response)
        
    except grpc.RpcError as e:
        # هندل کردن ارورهای gRPC
        if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
            print("خطا: زمان درخواست تمام شد (Timeout/Deadline Exceeded)")
        else:
            print(f"یک خطای دیگر رخ داد: {e.code()} - {e.details()}")

سمت سرور (Server-Side)#

این قسمت معمولاً توسط جونیورها فراموش می‌شود! وقتی کلاینت بعد از ۲ ثانیه بی‌خیال درخواست می‌شود، سرور شما نباید ۱۰ ثانیه درگیر پردازش آن بماند. سرور باید چک کند که آیا کلاینت هنوز منتظر است یا نه.

import time
import grpc

class MyServiceServicer(MyService_pb2_grpc.MyServiceServicer):
    def MyMethod(self, request, context):
        
        # کار طولانی ۱...
        time.sleep(1) 
        
        # الگوی Fail-Fast: چک می‌کنیم آیا زمان تمام شده یا کلاینت درخواست را کنسل کرده؟
        if not context.is_active():
            print("کلاینت بی‌خیال شده، منم پردازش رو متوقف می‌کنم.")
            return MyService_pb2.MyResponse()
            
        # چک کردن زمان باقیمانده (اختیاری ولی مفید برای لاگینگ یا پاس دادن به دیتابیس)
        print(f"زمان باقیمانده: {context.time_remaining()} ثانیه")

        # کار طولانی ۲...
        time.sleep(2)
        
        return MyService_pb2.MyResponse(result="Success")

۲. پترن‌ها و پرکتیس‌های (Best Practices) مدیریت Timeout#

حالا که کد بیسیک را دیدیم، بیایید معماری و پترن‌ها را بررسی کنیم:

الگو ۱: انتقال مهلت زمانی (Deadline Propagation)#

فرض کن سرویس A به سرویس B درخواست می‌دهد (با تایم‌اوت ۵ ثانیه). سرویس B برای جواب دادن باید به سرویس C درخواست بدهد. اشتباه رایج: سرویس B به سرویس C یک تایم‌اوت جدید مثلاً ۱۰ ثانیه‌ای بدهد! کار درست: سرویس B باید بررسی کند چقدر از آن ۵ ثانیه باقی مانده (مثلاً ۳ ثانیه) و همان زمان باقیمانده را به عنوان تایم‌اوت به C پاس بدهد.

در پایتون gRPC این کار معمولا از طریق گرفتن context.time_remaining() در سرور B و پاس دادن آن به عنوان timeout در کلاینت C انجام می‌شود.

الگو ۲: تلاش مجدد با تاخیر نمایی (Retry with Exponential Backoff)#

گاهی اوقات تایم‌اوت به خاطر یک مشکل لحظه‌ای در شبکه است. اگر بلافاصله یک ارور به کاربر نشان دهیم، تجربه کاربری بدی است. باید از پترن Retry استفاده کنیم. اما نه اینکه پشت سر هم رگبار درخواست بفرستیم! باید بین تلاش‌ها فاصله بیندازیم (مثلا تلاش اول ۱ ثانیه بعد، تلاش دوم ۲ ثانیه بعد، تلاش سوم ۴ ثانیه بعد). در پایتون می‌توانید از کتابخانه‌هایی مثل tenacity برای این کار استفاده کنید:

from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
import grpc

def is_deadline_exceeded(exception):
    return isinstance(exception, grpc.RpcError) and exception.code() == grpc.StatusCode.DEADLINE_EXCEEDED

# نهایتا ۳ بار تلاش کن، با تاخیر ۲، ۴ و ۸ ثانیه (فقط برای خطای تایم‌اوت)
@retry(stop=stop_after_attempt(3), 
       wait=wait_exponential(multiplier=2, min=2, max=10),
       retry=retry_if_exception_type(grpc.RpcError))
def robust_call(stub, request):
    try:
        return stub.MyMethod(request, timeout=2.0)
    except grpc.RpcError as e:
        if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
            raise e # بالا میندازیم تا tenacity دوباره تلاش کنه
        raise # بقیه خطاها رو مستقیما برگردون و ری‌ترای نکن

الگو ۳: مقدار جایگزین (Fallback / Graceful Degradation)#

گاهی اوقات دیتا حیاتی نیست. مثلاً در یک فروشگاه اینترنتی، سرویس “محصولات پیشنهادی” تایم‌اوت می‌دهد. آیا باید کل صفحه محصول لود نشود؟ خیر! پترن: اگر تایم‌اوت رخ داد، یک دیتای پیش‌فرض (Default Value)، دیتای کش شده (Cached) یا یک لیست خالی برگردانید و اجازه دهید بقیه سیستم کارش را بکند.

try:
    recommendations = stub.GetRecommendations(request, timeout=1.0)
except grpc.RpcError as e:
    if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
        # Fallback: برگرداندن لیست خالی یا محصولات پرفروش به صورت دیفالت
        recommendations = get_cached_or_default_recommendations()

الگو ۴: مدار شکن (Circuit Breaker)#

اگر سرویس B کلاً خراب شده باشد، هر درخواستی که به آن بفرستید بعد از مثلاً ۵ ثانیه تایم‌اوت می‌شود. اگر ۱۰۰۰ کاربر داشته باشید، سیستم شما ۵۰۰۰ ثانیه منابع را هدر می‌دهد تا منتظر یک سرویس خراب بماند! پترن Circuit Breaker (مدارشکن) می‌گوید: “اگر دیدی سرویس B تو ۵ درخواست اخیر همش تایم‌اوت داد، مدار رو قطع کن (باز کن). از این به بعد تا ۱ دقیقه اصلاً به سرویس B درخواست نفرست و بلافاصله خطای Fallback بده. بعد از ۱ دقیقه یک درخواست تستی بفرست ببین درست شده یا نه”. (برای پیاده‌سازی این پترن در پایتون پکیج‌هایی مثل pybreaker وجود دارند).

خلاصه کارهایی که باید به عنوان یک جونیور در شرکتتان انجام دهی:#

۱. در سمت کلاینت همیشه پارامتر timeout را ست کن. هیچ‌وقت درخواست بدون تایم‌اوت در پروداکشن نزن. ۲. خطای DEADLINE_EXCEEDED را try/except کن و برایش تصمیم بگیر (ری‌ترای کنم؟ ارور بدم؟ مقدار دیفالت بدم؟). ۳. در سمت سرور اگر حلقه (Loop) سنگین داری یا می‌خواهی کوئری سنگین دیتابیس بزنی، قبلش context.is_active() را چک کن که الکی منابع سرور را هدر ندهی.

موفق باشی! اگر بخشی از این مفاهیم نیاز به توضیح بیشتر داشت حتماً بپرس.

مهدی:

در مورد تایم اوت وسیع‌ترم کن

جمینای:

اینکه می‌خواهی دیدت را وسیع‌تر کنی، دقیقاً همان نقطه‌ای است که یک برنامه‌نویس Junior را به سمت Mid-level و Senior هل می‌دهد. در مهندسی نرم‌افزار، Timeout فقط یک پارامتر timeout=5 نیست؛ بلکه یک استراتژی دفاعی برای زنده نگه داشتن کل سیستم است.

برای اینکه دیدگاهت نسبت به تایم‌اوت وسیع و معماری‌محور (Architectural) شود، این ۶ مفهوم کلیدی را باید عمیقاً درک کنی:

۱. تفاوت Connection Timeout و Read/Write Timeout#

بسیاری از برنامه‌نویس‌ها فکر می‌کنند تایم‌اوت فقط یک زمان واحد است، در حالی که در شبکه‌ها (چه HTTP، چه gRPC و چه دیتابیس) ما معمولاً دو نوع تایم‌اوت داریم:

  • Connection Timeout (تایم‌اوت اتصال): چقدر حاضریم صبر کنیم تا اصلاً به سرور وصل شویم (مرحله Handshake و TCP). این زمان معمولاً باید خیلی کوتاه باشد (مثلاً ۱ تا ۳ ثانیه). اگر سروری در ۳ ثانیه اجازه اتصال ندهد، احتمالاً کلاً قطع است.
  • Read Timeout (تایم‌اوت خواندن/پردازش): حالا که وصل شدیم و درخواست را فرستادیم، چقدر حاضریم برای دریافت جواب صبر کنیم؟ این زمان به منطق بیزینس (Business Logic) بستگی دارد. مثلاً پردازش تصویر شاید ۱۰ ثانیه زمان بخواهد.

(نکته عملی: در کتابخانه requests پایتون، حرفه‌ای‌ها تایم‌اوت را به صورت Tuple می‌دهند: timeout=(3, 10) یعنی ۳ ثانیه برای اتصال، ۱۰ ثانیه برای خواندن جواب).

۲. مفهوم Idempotency (آیدم‌پوتِنسی) و خطر مرگبار Retry#

فرض کن سرویس A به سرویس B (سیستم پرداخت) می‌گوید: “۵۰ هزار تومان از حساب کاربر ۱۲۳ کم کن”. بعد از ۵ ثانیه تایم‌اوت می‌شود. حالا سرویس A چه می‌داند؟ آیا درخواست اصلاً به B نرسید؟ یا به B رسید، پول کم شد، اما در مسیرِ برگشتِ جواب، شبکه قطع شد (تایم‌اوت)؟

اگر از الگوهایی که قبلاً گفتم (Retry) استفاده کنی و دوباره درخواست بفرستی، ممکن است ۱۰۰ هزار تومان از کاربر کم شود! فاجعه! راه حل معماری: سرویس‌ها باید Idempotent باشند. یعنی هر درخواست باید یک شناسه یکتا (Idempotency Key) داشته باشد. اگر سرویس B درخواستی با همان شناسه را دوباره دریافت کرد، نباید دوباره پول کم کند، بلکه فقط باید جواب موفقیت‌آمیز قبلی را برگرداند. قانون طلایی: فقط زمانی مجاز به استفاده از Retry در خطاهای تایم‌اوت هستی که بدانی متد مقصد Idempotent است.

۳. خرابی آبشاری (Cascading Failures) و فاجعه کمبود منابع#

تایم‌اوت برای جلوگیری از “خرابی آبشاری” است. تصور کن ۳ میکروسرویس داریم: A -> B -> C

  1. سرویس C به مشکل می‌خورد و جواب نمی‌دهد.
  2. اگر سرویس B تایم‌اوت نداشته باشد، تمام درخواست‌هایی که به C می‌فرستد در وضعیت “انتظار” باقی می‌مانند (Hang می‌کنند).
  3. وقتی درخواست‌ها در انتظار بمانند، Thread ها (یا Worker ها) و Memory سرویس B پر می‌شود.
  4. حالا سرویس B کلاً از کار می‌افتد (Resource Exhaustion).
  5. حالا سرویس A که به B متصل بود هم Worker هایش پر می‌شود و از کار می‌افتد.
  6. کل سیستم سقوط می‌کند!

تایم‌اوت در واقع “شیر اطمینان” است. می‌گوید: قبل از اینکه منابع من پر شود، بی‌خیال این درخواست می‌شوم تا بتوانم به بقیه مشتری‌ها سرویس بدهم.

۴. تایم‌اوت‌ها کجا پنهان شده‌اند؟ (نشت تایم‌اوت در لایه‌های مختلف)#

یک سیستم فقط کد gRPC شما نیست. یک درخواست از لایه‌های مختلفی رد می‌شود و هر لایه تایم‌اوت خودش را دارد. اگر این‌ها با هم تنظیم نباشند، دیباگ کردنش کابوس است:

  • Load Balancer ها (مثل Nginx, HAProxy, Envoy): اگر Nginx روی ۶۰ ثانیه تایم‌اوت تنظیم شده باشد ولی gRPC شما روی ۱۲۰ ثانیه، کاربر بعد از ۶۰ ثانیه خطای Nginx (504 Gateway Timeout) می‌گیرد در حالی که سرور gRPC شما هنوز دارد در پس‌زمینه کار می‌کند!
  • Database (Connection Pool): آیا در پایتون (مثلاً SQLAlchemy یا موتور دیتابیس) تنظیم کرده‌ای که کوئری حداکثر چقدر طول بکشد؟ (Statement Timeout).
  • DNS Lookup: گاهی تایم‌اوت به خاطر این است که سرور نمی‌تواند آدرس IP مقصد را پیدا کند.

۵. چطور مقدار تایم‌اوت را تعیین کنیم؟ (SLI/SLA و Percentiles)#

برنامه‌نویس‌های تازه‌کار عدد تایم‌اوت را حسی انتخاب می‌کنند (مثلاً می‌گویند “خب ۵ ثانیه خوبه دیگه!”). در سطح بالاتر، ما از متریک‌ها (Metrics) استفاده می‌کنیم: شما باید به داشبورد مانیتورینگ (مثل Grafana/Prometheus) نگاه کنید. معیاری داریم به نام P99 (صدک ۹۹). یعنی ۹۹ درصد درخواست‌های سیستم من در چه زمانی پردازش می‌شوند؟ اگر P99 شما ۲۰۰ میلی‌ثانیه است، یعنی تقریباً همه در ۲۰۰ms کارشان تمام می‌شود. پس گذاشتن تایم‌اوتِ ۱۰ ثانیه احمقانه است! تایم‌اوت در این حالت باید مثلاً ۵۰۰ میلی‌ثانیه یا ۱ ثانیه باشد. هرچه تایم‌اوت کوتاه‌تر باشد، سیستم سریع‌تر متوجه خرابی می‌شود و منابع آزاد می‌شوند.

۶. مشکل ردیابی توزیع شده (Distributed Tracing)#

وقتی یک معماری میکروسرویس بزرگ دارید (مثلا ۲۰ سرویس که به هم وصلند)، کاربر نهایی خطای Timeout می‌گیرد. شما از کجا می‌فهمید کدام سرویس در این زنجیره مقصر بوده است؟ اینجا مفهومی به نام Trace ID وارد می‌شود. در بدو ورود درخواست به سیستم، یک ID یکتا ساخته می‌شود (توسط ابزارهایی مثل Jaeger, Zipkin یا OpenTelemetry). این ID از طریق Metadata در gRPC به تمام سرویس‌ها پاس داده می‌شود. اگر تایم‌اوتی رخ دهد، شما در لاگ‌ها آن Trace ID را سرچ می‌کنید و دقیقاً می‌بینید گلوگاه (Bottleneck) در کدام میکروسرویس و حتی کدام کوئری دیتابیس بوده است.

خلاصه دیدگاه وسیع (Big Picture):#

به عنوان یک مهندس نرم‌افزار، از این به بعد وقتی کلمه Timeout را می‌شنوی، نباید فقط یاد کد try/except بیفتی. باید از خودت بپرسی:

  1. “اگر اینجا تایم‌اوت شود، آیا سیستم می‌تواند خودش را جمع‌وجور کند؟”
  2. “آیا تکرار این کار بعد از تایم‌اوت، دیتابیس ما را کثیف (Inconsistent) نمی‌کند؟”
  3. “آیا تایم‌اوتی که ست کرده‌ام، با منطق بیزینس و تایم‌اوت بقیه سیستم همخوانی دارد؟”

اگر این سوالات را در جلسات طراحی سیستم شرکت بپرسی، مطمئن باش روی تو به عنوان یک مهندس پخته و آینده‌دار حساب خواهند کرد!

مهدی:

من اسم OpenTelemetry رو زیاد شنیدم ولی نمی‌دونم چیه و وظیفه‌اش چیه. بهم یاد بده

جمینای:

بسیار عالی! رسیدیم به یکی از مهم‌ترین و مدرن‌ترین مفاهیم مهندسی نرم‌افزار. برای درک OpenTelemetry (یا به اختصار OTel)، بیایید ابتدا مشکلی که این ابزار حل می‌کند را با یک مثال روزمره بررسی کنیم.

مشکل چیست؟ (معمای قتل در میکروسرویس‌ها!)#

تصور کن در یک سیستم یکپارچه (Monolith) برنامه‌نویسی می‌کنی. یک ارور رخ می‌دهد، لاگ‌ها را باز می‌کنی، شماره خط کد را می‌بینی و مشکل را حل می‌کنی. ساده است!

حالا سیستم شرکت شما میکروسرویس و gRPC است. کاربر روی دکمه “ثبت سفارش” کلیک می‌کند. درخواست او این مسیر را طی می‌کند: گیت‌وی (Gateway) -> سرویس سفارش (A) -> سرویس انبار (B) -> سرویس پرداخت (C)

ناگهان کاربر خطای Timeout می‌گیرد! (همان مبحث قبلی خودمان). حالا شما به عنوان برنامه‌نویس باید بفهمی مشکل از کجا بوده. آیا سرویس B کُند شده؟ آیا دیتابیسِ سرویس C قفل شده؟ آیا شبکه بین A و B قطع شده؟ بدون ابزار مناسب، پیدا کردن مقصر مثل حل کردن یک معمای قتل است! باید به سرورهای مختلف وصل شوی و هزاران خط لاگ متنی را بالا و پایین کنی به این امید که سرنخی پیدا کنی.

OpenTelemetry چیست؟#

اوپن‌تله‌متری (OpenTelemetry) یک استاندارد و مجموعه‌ای از ابزارها (SDK) است که به ما کمک می‌کند بفهمیم “درون سیستم ما چه می‌گذرد”. به این قابلیت در مهندسی نرم‌افزار Observability (رصدپذیری) می‌گویند.

⚠️ یک اشتباه بسیار رایج: خیلی‌ها فکر می‌کنند OTel یک دیتابیس یا داشبورد گرافیکی (مثل Grafana یا Kibana) است. خیر! OpenTelemetry فقط وظیفه تولید، جمع‌آوری و ارسال داده‌ها را بر عهده دارد. اینکه این داده‌ها کجا ذخیره شوند و چطور نمایش داده شوند، به عهده ابزارهای دیگر (مثل Jaeger, Prometheus, Datadog) است. OTel به شما می‌گوید: “داده‌ها را استاندارد کن و به هر نرم‌افزاری که دوست داری بفرست.”

OpenTelemetry دقیقاً چه چیزی را جمع‌آوری می‌کند؟#

OTel بر پایه سه ستون اصلی رصدپذیری (Three Pillars of Observability) بنا شده است:

۱. Traces (ردیابی‌ها - مهم‌ترین بخش برای میکروسرویس‌ها)#

تِریس (Trace) مثل کد رهگیریِ بسته‌های پستی است. وقتی درخواست کاربر وارد سیستم می‌شود، OTel یک Trace ID یکتا (مثلاً id: 8f92a) به آن اختصاص می‌دهد. این ID در هدرهای gRPC قرار می‌گیرد و به تمام سرویس‌ها (A به B به C) پاس داده می‌شود. در هر سرویس، یک Span (بازه زمانی) ایجاد می‌شود. مثلاً:

  • Span 1: سرویس A درخواست را گرفت (زمان کل: ۵ ثانیه)
  • Span 2: سرویس A به سرویس B درخواست داد (زمان: ۱ ثانیه)
  • Span 3: سرویس A به سرویس C درخواست داد (زمان: ۳.۹ ثانیه) -> آها! فهمیدیم کندی از سرویس C بوده است!

۲. Metrics (متریک‌ها / معیارها)#

متریک‌ها اعداد و ارقام کلی هستند که وضعیت سلامت سیستم را نشان می‌دهند. (همان P99 که در بحث تایم‌اوت گفتم). مثلاً:

  • چند درصد پردازنده (CPU) درگیر است؟
  • در یک دقیقه گذشته چند خطای تایم‌اوت داشتیم؟
  • میانگین زمان پاسخگویی سرویس پرداخت چقدر است؟

۳. Logs (لاگ‌ها)#

همان متون ساده‌ای که با print یا logger.info در پایتون چاپ می‌کنید. هنر OTel این است: لاگ‌ها را به Trace ID متصل می‌کند. یعنی شما می‌گویید “لاگ‌های مربوط به درخواست 8f92a را در تمام میکروسرویس‌ها به من نشان بده”. اینطوری فقط لاگ‌های همان یک کاربر خاص را در کل سیستم می‌بینید!

معماری کار با OpenTelemetry چگونه است؟#

روند کار به این شکل است:

  1. کد شما (مثلا Python + gRPC): شما کتابخانه OTel را در پایتون نصب می‌کنید.
  2. Instrument کردن (تجهیز کردن): OTel به صورت خودکار (Auto-instrumentation) می‌فهمد که شما دارید از gRPC استفاده می‌کنید. بدون اینکه شما هزار خط کد بنویسید، خودش Trace ID ها را می‌سازد و به درخواست‌های gRPC می‌چسباند.
  3. Collector (جمع‌آوری کننده): کدهای شما داده‌ها را به یک برنامه جانبی به نام OTel Collector می‌فرستند. این برنامه داده‌ها را فیلتر، تمیز و فشرده می‌کند.
  4. Backend (مقصد نهایی): حالا Collector داده‌ها را می‌فرستد به داشبوردهای شرکت شما. مثلا Traceها را به Jaeger می‌فرستد تا شما به صورت گرافیکی (نمودارهای درختی) ببینید هر درخواست چقدر طول کشیده است.

چرا OTel برای شما (در پایتون و gRPC) یک نعمت است؟#

اگر OTel نبود، شما برای پاس دادن Trace ID بین میکروسرویس‌ها باید در تک تک متدهای gRPC کدهای زیر را دستی می‌نوشتید:

  • “آی‌دی را از Context بخوان.”
  • “زمان شروع را ثبت کن.”
  • “آی‌دی را در Context درخواستِ جدید قرار بده.”
  • “زمان پایان را ثبت کن.”
  • “اختلاف زمان را حساب کن و بفرست.”

اما در پایتون، با استفاده از قابلیت‌های OTel و Interceptors در gRPC، شما فقط با اضافه کردن چند خط کد در زمان اجرای سرور و کلاینت، تمام این اتفاقات را به صورت اتوماتیک رقم می‌زنید (به این کار Auto-Instrumentation می‌گویند).

خلاصه#

OpenTelemetry یک زبان مشترک است تا کدهای پایتون، گو (Go)، جاوا و… در سیستم شما بتوانند اطلاعاتِ “مدت زمان اجرا”، “خطاها” و “آمار سیستم” را با یک فرمت استاندارد تولید کنند. به کمک OTel، وقتی دفعه بعد سیستم تایم‌اوت داد، به جای حدس زدن، داشبورد را باز می‌کنی و دقیقاً می‌بینی کدام میکروسرویس و حتی کدام کوئری دیتابیس باعث این تاخیر شده است.

مهدی:

در مورد Interceptors بهم توضیح بده

جمینای:

رسیدیم به یکی از جذاب‌ترین و کاربردی‌ترین ابزارهای gRPC! در واقع، Interceptor (اینتِرسِپتور - به معنی رهگیر یا متوقف‌کننده) همان قطعه گمشده‌ای است که باعث می‌شود OpenTelemetry به صورت جادویی کار کند و نیازی نباشد شما کدهای تکراری بنویسید.

برای درک Interceptor، بیایید اول مشکلی که حل می‌کند را بررسی کنیم.

مشکل: تکرار کدهای خسته‌کننده (نقض قانون DRY)#

فرض کن در سرور gRPC خود ۱۰ تا متد (Service) مختلف داری. مدیرت از تو می‌خواهد که: ۱. قبل از اجرای هر متد، توکن اعتبارسنجی (JWT) کاربر چک شود. ۲. زمان شروع و پایان هر متد در کنسول لاگ (Log) شود. ۳. اگر خطایی رخ داد، لاگ شود.

راه حل جونیورها: رفتن به داخل تک‌تک این ۱۰ متد و نوشتن کدهای if token_is_valid و start_time = time.time(). نتیجه: کدهایت به شدت شلوغ می‌شود و اگر فردا یک متد یا شرط جدید اضافه شود، باید همه جا را تغییر دهی!

راه حل: Interceptor چیست؟#

در دنیای وب (مثل فریم‌ورک‌های Django یا FastAPI) به این مفهوم Middleware می‌گویند. در gRPC به آن Interceptor می‌گویند.

مثال دنیای واقعی: Interceptor مثل گیت بازرسی فرودگاه است. شما به جای اینکه درِ ورودی هر هواپیما یک مامور بگذارید که مسافران را بگردد، یک گیت اصلی می‌گذارید. هر درخواستی که می‌خواهد به متدهای شما (هواپیماها) برسد، باید اول از این گیت (Interceptor) رد شود.

Interceptor به شما اجازه می‌دهد کدی را بنویسید که “قبل” و “بعد” از متد اصلی شما اجرا شود، بدون اینکه کد اصلی دست بخورد.

کاربردهای اصلی Interceptors#

  1. احراز هویت (Authentication & Authorization): چک کردن اینکه آیا کلاینت اجازه دسترسی دارد یا نه. اگر نداشت، Interceptor در همان گیت فرودگاه درخواست را ریجکت می‌کند (خطای UNAUTHENTICATED می‌دهد) و اصلاً درخواست به متد شما نمی‌رسد!
  2. لاگینگ و مانیتورینگ (مثل OpenTelemetry): زمان ورود درخواست را ثبت می‌کند، متد اجرا می‌شود، زمان خروج را ثبت می‌کند و اختلاف این دو را به عنوان زمان پردازش به داشبورد می‌فرستد.
  3. مدیریت تایم‌اوت و Retry: (ارتباط با سوالات قبلی‌ات!) می‌توانی Interceptor ای در سمت کلاینت بنویسی که بگوید: “اگر برنامه‌نویس یادش رفت تایم‌اوت بگذارد، به صورت خودکار روی تمام درخواست‌ها تایم‌اوت ۵ ثانیه‌ای اعمال کن.”
  4. هندل کردن ارورهای گلوبال: تبدیل خطاهای دیتابیس به خطاهای استاندارد gRPC قبل از ارسال به کاربر.

پیاده‌سازی Interceptor در پایتون#

در gRPC ما دو نوع Interceptor داریم: سمت سرور (درخواست‌های ورودی را رهگیری می‌کند) و سمت کلاینت (درخواست‌های خروجی را رهگیری می‌کند).

بیایید یک مثال واقعی از Interceptor سمت سرور برای لاگ کردن زمان اجرای متدها بنویسیم:

۱. نوشتن کلاس Interceptor#

import grpc
import time

# باید از این کلاس ارث‌بری کنیم
class LoggingInterceptor(grpc.ServerInterceptor):
    
    def intercept_service(self, continuation, handler_call_details):
        # ----------------------------------------------------
        # بخش ۱: کارهایی که "قبل" از رسیدن به متد اصلی انجام میدهیم
        # ----------------------------------------------------
        method_name = handler_call_details.method
        print(f"[Interceptor] درخواست دریافت شد برای متد: {method_name}")
        start_time = time.time()

        # ----------------------------------------------------
        # بخش ۲: پاس دادن درخواست به مرحله بعد (یا همون متد اصلی شما)
        # ----------------------------------------------------
        # کلمه continuation یعنی "ادامه بده". اینجا متد اصلی شما اجرا میشه
        response_or_iterator = continuation(handler_call_details)

        # ----------------------------------------------------
        # بخش ۳: کارهایی که "بعد" از اتمام متد اصلی انجام میدهیم
        # ----------------------------------------------------
        duration = time.time() - start_time
        print(f"[Interceptor] پردازش متد {method_name} در {duration:.4f} ثانیه تمام شد.")
        
        # جواب را برمی‌گردانیم تا به دست کلاینت برسد
        return response_or_iterator

۲. اضافه کردن Interceptor به سرور#

حالا فقط کافیست وقتی سرور gRPC را استارت می‌کنی، این گیت بازرسی را به آن معرفی کنی:

from concurrent import futures
import grpc

def serve():
    # اینترسپتور را نمونه‌سازی می‌کنیم
    my_interceptor = LoggingInterceptor()
    
    # موقع ساخت سرور، آن را پاس می‌دهیم
    server = grpc.server(
        futures.ThreadPoolExecutor(max_workers=10),
        interceptors=(my_interceptor,)  # <-- جادوی کار اینجاست!
    )

ارتباط Interceptor با OpenTelemetry#

حالا که فهمیدی Interceptor چیست، درک کار OpenTelemetry برایت مثل آب خوردن می‌شود!

وقتی شما کتابخانه OTel را در پایتون نصب می‌کنید، در واقع نیازی نیست خودتان دستی کدی بنویسید. تیم سازنده OTel قبلاً یک کلاس Interceptor (مثل OpenTelemetryClientInterceptor و OpenTelemetryServerInterceptor) نوشته‌اند.

این Interceptor در پس‌زمینه کارهای زیر را می‌کند:

  • در سمت کلاینت: قبل از ارسال درخواست، Trace ID را می‌سازد و به Metadata (معادل Header در HTTP) اضافه می‌کند.
  • در سمت سرور: Metadata را می‌خواند، Trace ID را استخراج می‌کند (تا بفهمد این درخواست ادامه کدام درخواست قبلی است)، زمان شروع و پایان را ثبت می‌کند و به صورت خودکار برای سیستم جمع‌آوری داده می‌فرستد.

یک نکته حرفه‌ای (Pro Tip) برای جونیورها:#

در gRPC، متدها می‌توانند Unary (یک درخواست، یک جواب) یا Streaming (جریان داده) باشند. در پایتون، نوشتن Interceptor برای متدهای Streaming کمی پیچیده‌تر است. پکیجی به نام grpc-interceptor در پایتون وجود دارد که توسط کامیونیتی ساخته شده و نوشتن Interceptorها (مخصوصا برای مدیریت ارورها) را به شدت تمیزتر و راحت‌تر می‌کند. پیشنهاد می‌کنم در پروژه‌های واقعی شرکتتان، نگاهی به این پکیج هم بیندازی.
